Introduction

From the previous lecture we already know that R allows different types of data and that we may build collections with data like vectors matrices and arrays. However, these data collections we learn so far only accept elements of the same time.

Most of our databases have mixed data types: numbers, words (strings), categorical data (factors). How can you deal with that in R?

R programming language makes a distinction between objects that are ATOMIC (object of the same mode, like vector) and those that are RECURSIVE (that might contain multiple modes).

We will look first to data.frames, than we will see lists and at the end some special data structures (and data.frames again!).

Framing dataframes

Dataframes are table-like structures that can hold different types of data. They allow access to data in similar ways as matrices.

?data.frame

There are multiple ways to build a dataframe. Let’s start by a set of separate vectors with different types:

vec1 <- c(1, 2, 3, 4, 5)
vec3 <- rnorm(5)
vec5 <- c("male", "male", "female", "female", "female")

df1 <- data.frame(vec1, vec3, vec5)
df1
##   vec1       vec3   vec5
## 1    1 -1.6660431   male
## 2    2 -0.3184210   male
## 3    3  1.2815752 female
## 4    4 -0.3112347 female
## 5    5  0.6395818 female

We could also use a matrix:

mat <- matrix(1:16, nrow=4, byrow=T)
df2 <- as.data.frame(mat)
df2
##   V1 V2 V3 V4
## 1  1  2  3  4
## 2  5  6  7  8
## 3  9 10 11 12
## 4 13 14 15 16

However, if all our data is numeric, it would be better to keep it as a matrix.

Notice that on ‘df1’ above, the columns inherited the names of the vectors and on ‘df2’ we got V(number) names.

We easily change column and row names of a dataframe.

colnames(df1) <- c("var1", "var2", "var3")
df1
##   var1       var2   var3
## 1    1 -1.6660431   male
## 2    2 -0.3184210   male
## 3    3  1.2815752 female
## 4    4 -0.3112347 female
## 5    5  0.6395818 female

Be careful because names are strings: if you provide a numeric value to column names it will be coerced to character type and may be the source of many errors (more in the indexing class).

We can also name columns directly when building a dataframe:

df1 <- data.frame(ID=vec1, value=vec3, sex=vec5)
df1

Lists

Lists are recursive objects: can hold any type of data including their own type. They are a basal object in most complex R code but seldom used in common scripts.

?list

So let’s start creating a list

myemptylist <- list() 
myemptylist
## list()

we can also create lists with some data. In the following example we create a list holding different data types:

my.list1 <- list(vec1, vec3, 23, "this is a list")
my.list1
## [[1]]
## [1] 1 2 3 4 5
## 
## [[2]]
## [1] -1.6660431 -0.3184210  1.2815752 -0.3112347  0.6395818
## 
## [[3]]
## [1] 23
## 
## [[4]]
## [1] "this is a list"

Notice that the way R displays data in list is different than what we saw before. Elements in lists are ordered with [[]] and inside each element we have the data we created with the associated data type.

Other special data structures

Although data.frames are very tangled with R, being used in most scripts, are they an object of their own? We will see that, but first let’s have a look to what we are calling special objects.

Let’s create a simple vector with three elements:

mypoints <- c(1, 2, 3)
names(mypoints) <- c("A", "B", "C")

As you see, these points can be represented in a line:

Let’s manually build a matrix of the Euclidean distance between these points:

mydistMat <- matrix(c(0,1,2,1,0,1,2,1,0), 3, 3)
colnames(mydistMat) <- names(mypoints)
rownames(mydistMat) <- names(mypoints)
myDistMat
## Error in eval(expr, envir, enclos): object 'myDistMat' not found

What is redundant in this representation? Is there any other way to represent the distance matrix?

  1. The diagonal is ZERO
  2. The matrix is symetric.

We can use an R function to build this distance matrix and have a special object:

mydist <- dist(mypoints)
mydist
##   A B
## B 1  
## C 2 1

Let’s check some of the proprieties of the newly created object:

mode(mydist)
## [1] "numeric"
typeof(mydist)
## [1] "double"
class(mydist)
## [1] "dist"

This is showing us is that the only difference to the other objects (vector and matrix) is the class. The class is the propriety that makes different object behave differently:

  • they have different ways to print their contents
  • they might do a special plot
  • they might be converted to other classes.
as.matrix(mydist)
##   A B C
## A 0 1 2
## B 1 0 1
## C 2 1 0

An important thing to keep in mind is that classes can be defined by the user. This means that new packages can have their own classes and their own methods to deal with specific data formats. Although it makes a little be more difficult to understand the code, the use of those classes became much more simple. We don’t really need to see what the code is, we just have to know what it does! For instance, let’s open a tree in R, using library ape:

library(ape)
tree <- read.tree("randomTree.tre")

Now you would expect a list of things like nodes, tip names, values, etc or a text description of a tree like the Newick format…

tree
## 
## Phylogenetic tree with 100 tips and 99 internal nodes.
## 
## Tip labels:
##  t81, t85, t1, t91, t28, t45, ...
## 
## Rooted; includes branch lengths.

This means that tree has a dedicated class and that class has methods defined for showing what is inside.

class(tree)
## [1] "phylo"
mode(tree)
## [1] "list"
typeof(tree)
## [1] "list"

An object of class phylo is simply a list! Typical generic function for classes are the summary:

sumamry(tree)
## Error in sumamry(tree): could not find function "sumamry"

or a plot

plot(tree)
